# 第一阶段 MVP

## 一句话目标

先做出一个“本地优先、可长期记忆、可接入授权知识、具备基础工具能力”的开放智能体原型。

## 为什么是这个 MVP

如果第一版连以下事情都做不到：

- 记得住
- 接得上
- 跑得起来
- 可被别人继续开发

那就没有资格继续谈更大的愿景。

所以第一阶段必须非常克制。

## 第一阶段必须有的能力

### 1. 本地部署

要求：

- 能在一台普通开发机器上启动
- 部署方式尽量简单
- 最好使用 Docker

验收：

- 新加入的人能按说明跑起来

### 2. 长期记忆

要求：

- 至少有一层长期记忆
- 能保存用户重要信息
- 能在会话结束后继续保留

验收：

- 关闭后重开仍然保留关键记忆

### 3. 授权知识接入

要求：

- 能导入本地文档
- 能进行基础检索
- 能引用知识而不是纯凭模型猜

验收：

- 对同一问题，能结合导入知识给出回答

### 4. 基础编排

要求：

- 能根据请求决定是否查记忆、查知识、调用工具
- 不要求复杂自治

验收：

- 至少有一条清晰的工作流跑通

### 5. 有边界的工具调用

要求：

- 能做少量基础操作
- 不能默认开放全部权限
- 高风险动作要可控

验收：

- 能展示受限工具调用能力

### 6. 审计与恢复

要求：

- 有日志
- 有配置
- 有备份思路

验收：

- 基础问题可追踪
- 核心数据可以恢复

## 第一阶段不要做的事

以下事项不要混进 MVP：

- 全网分布式永久存续
- 全自动学习互联网全部知识
- 全球治理机制
- 区块链化
- 代币系统
- 复杂多智能体生态
- “数字生命”级叙事实现

这些会直接把第一阶段拖死。

## 推荐拆分模块

### 模块 A：本地运行壳子

- 配置
- 启动
- Docker
- 基础 API

### 模块 B：记忆系统

- 短期记忆
- 长期记忆
- 更新策略

### 模块 C：知识系统

- 文档导入
- 索引
- 检索

### 模块 D：编排系统

- 意图判断
- 路由
- 工具调用控制

### 模块 E：审计系统

- 日志
- 错误记录
- 行为记录

## 第一批 issue 建议

- 选型本地部署方式
- 设计长期记忆结构
- 设计知识导入流程
- 设计最小工作流
- 设计工具权限模型
- 设计日志与审计基础
- 做出第一版前端壳子

## 第一阶段完成的标志

当下不以“多强”为标准，而以“能不能继续做”为标准。

如果出现以下结果，第一阶段就算成功：

- 新人能跑起来
- 用户能看到长期记忆效果
- 授权知识接入有效
- 工具调用是受控的
- 代码结构足够清楚
- 有人愿意继续加入一起做
