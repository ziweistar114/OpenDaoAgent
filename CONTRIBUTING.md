# Contributing

感谢你对 `OpenDaoAgent` 感兴趣。

这个项目当前仍处在早期发起阶段，所以我们比起“快速堆功能”，更重视以下几件事：

- 方向不要跑偏
- 边界要清楚
- 模块要能继续长
- 协作要能持续

## 当前更欢迎什么贡献

目前更欢迎这几类贡献：

### 1. 架构贡献

- 长期记忆设计
- 本地优先部署方案
- 跨设备连续性方案
- 工具调用边界设计
- 审计与回滚机制设计

### 2. 工程贡献

- 后端基础能力
- Agent 编排
- RAG / 检索 / 知识接入
- Docker / 本地部署
- 最小可运行原型

### 3. 产品与文档贡献

- README 优化
- MVP 范围收敛
- 用户路径梳理
- 招募与社区文案
- issue 模板、讨论模板

## 贡献前建议

在提交代码或文档前，请先确认：

1. 这项改动是否服务于当前阶段目标
2. 它是否扩大了不必要的范围
3. 它是否破坏了本地优先、长期记忆、可审计这几个核心原则
4. 它是否会把项目推向“看起来很大，但无法交付”

## 当前不鼓励的方向

以下方向暂时不鼓励优先投入：

- 区块链化
- DAO 治理实现
- Token 经济
- 超前 AGI 叙事
- 复杂全球分布式架构
- 过度自动化和无边界自治

这些方向并不一定永远不做，但不属于当前阶段的重点。

## 协作建议

如果你想加入，推荐顺序是：

1. 先读 `README.md`
2. 再读 `MANIFESTO.md`
3. 再看 `docs/FIRST_MVP.md`
4. 认领 issue 或先发 discussion

## Issue 建议标签

建议首批 issue 使用这些标签：

- `good first issue`
- `architecture`
- `memory`
- `rag`
- `local-first`
- `infra`
- `frontend`
- `docs`
- `discussion needed`

## 提交风格

我们更欢迎：

- 小步、清楚、可解释的提交
- 能说明“为什么这么改”的提交
- 不夸大、不偷换概念的文档

我们不欢迎：

- 一次性大而杂的改动
- 没有边界说明的功能扩张
- 把愿景文案写成已实现事实

## 最后

这个项目当前更像一个长期方向的“起点工程”，而不是成熟产品。

所以最重要的不是显得多厉害，而是：

- 真的能跑
- 真的能记
- 真的能继续做下去
